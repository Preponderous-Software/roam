Frontend-abstraction epic · #433

Roam runs on interfaces, not pygame

Game and screen logic depend on three small backend-neutral interfaces — Renderer, InputSource, and Clock — plus a handful of value types. Swap the backend behind them and the same game drives a different frontend, with zero changes to game logic.

01The core idea

Before the epic, every screen called pygame directly — for drawing and input. Now the dependency arrow is inverted: logic points at an interface; a backend implements it.

The abstraction is not just asserted — it's proven. A test poisons pygame in sys.modules, then drives the Screen run loop one full frame through the Null frontend. If any module on that path imported pygame, the subprocess dies with ImportError instead of printing its sentinel. Game logic genuinely doesn't touch pygame.

03The interfaces, briefly

The full surface a frontend implements. Small on purpose.

Renderer output

  • Lifecycle: getDisplaySize / Width / Height, clearScreen(color), present(), setCaption(text), getGameAreaRect()
  • Primitives: drawRectangle, drawText, drawTextLeftAligned, drawButton, drawTranslucentOverlay(rgba), drawImage
  • Images: loadImage (cached, placeholder-on-miss), scaleImage, createSurface, saveImage, tryLoadImage (uncached, None-on-miss)
  • Targets & misc: getRenderTarget / setRenderTarget (offscreen render), setClipRegion, captureScreenshot

InputSource input

  • pollEvents() → a list of neutral InputEvents (replaces pygame.event.get())
  • isPressed(keyCode), getMousePosition(), getMouseButtons() — current device state
  • Events carry an EventType and a backend-neutral KeyCode; the backend maps native input at the boundary.

Clock timing

  • tick(fps) — pace the loop to at most fps/sec; returns elapsed ms. The pygame backend wraps pygame.time.Clock; a headless one never blocks.

04Add a frontend in three steps

No game-logic changes — only new files plus one factory branch.

Implement the three interfaces

An XRenderer(Renderer), XInputSource(InputSource), and XClock(Clock) against your backend's primitives. The renderer-contract test checks every method screens call is on the interface.

Build them in a frontend factory

Add an XFrontend exposing getRenderer()/getInputSource()/getClock(), and a branch in createFrontend(config) selecting it.

Register the instances in DI

roam.py registers the three with the container, exactly as it does the pygame ones today. Screens auto-wire the interfaces — nothing else moves.

05What survives, what degrades

An honest map: not every effect is meaningful on every backend, and that's fine — the seam lets a backend skip what it can't express.

  • Universal: the run loop, screen transitions, input dispatch, layout math, text, rectangles — every backend does these.
  • Images: native on Pygame/SDL/Web; on Text they become a glyph or label; Null no-ops.
  • Day/night light masks: per-pixel alpha + blend compositing — intrinsically pixel-graphics. Native on Pygame/SDL/Web; Text/Null skip it. This is the one effect deliberately left as a documented pygame exception in the live renderer rather than forced behind a leaky primitive.
  • Frame pacing & screenshots: trivially per-backend (or no-op for headless/text).
